# Brief: Handoff CLI Tool

## 2026-02-20: Status subcommand + scope expansion

**Context:** Main session discussion after merging worktree-rm-fixes.

### Status output should be scripted

STATUS mode (execute-rule.md MODE 1) is pure data transformation — no judgment. Inputs:
- session.md parsing (pending tasks, worktree tasks, metadata)
- `_worktree ls` output
- Plan-to-task cross-referencing
- Parallel group detection (rule-based)

Currently generated by the agent each time (`s`, post-commit in `ci`/`hc`). Expensive, inconsistent formatting, no tests.

**Decision:** Script it as a CLI subcommand. Shared parsing layer with handoff operations.

### Internal CLI convention

These commands are skill-internal — invoked by skills/agents, never by the user directly. Follow `_worktree` convention:
- Underscore prefix: `_status`, `_handoff`, `_commit` (or grouped under `_session`)
- Hidden from `--help`
- No namespace pollution concern

The skills (`/handoff`, `/commit`, execute-rule STATUS) remain the user interface.

### Scope adjustment

Original scope: mechanical handoff+commit pipeline.
Expanded scope: status output as a subcommand within the same CLI infrastructure. Shared session.md parsing layer serves both status display and handoff operations.

Whether flat (`_status`, `_handoff`) or grouped (`_session status`, `_session handoff`) depends on shared infrastructure — design decision for outline.

## 2026-03-08: Markdown AST parser ordering consideration

**Source:** bootstrap-tag-support post-execution review.

The S-4 session.md parser (shared parser in `src/claudeutils/session/parse.py`) is designed as regex-based line-by-line parsing. A new plan (`plans/markdown-ast-parser/brief.md`) proposes a two-stage architecture: preprocessor (fix LLM errors) → standard parser (AST) → domain traversals.

**Ordering decision needed:**
- **AST-first (prerequisite):** Build S-4 parser directly on AST from the start. Avoids throwaway regex parser. Requires markdown-ast-parser plan to ship first (at least the parser utility layer).
- **Regex-first (independent):** Build S-4 with regex per current design. markdown-ast-parser becomes a follow-up rewrite. Ships faster but creates technical debt.

**Impact:** S-4 parser is shared infrastructure — handoff, status, and validation all consume it. Building on AST from the start means all consumers inherit structural correctness. Building on regex means rewriting 3 consumers later.

**Blocker status:** Bootstrap tag support task complete — runbook generation unblocked. Next step: regenerate step files via `prepare-runbook.py plans/handoff-cli-tool/`, then orchestrate. The parser ordering decision affects S-4 design, not the overall plan structure.

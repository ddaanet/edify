## Summary

Three prose artifacts reviewed against the design specification across 10 review axes (conformance, functional correctness, functional completeness, vacuity, excess, actionability, constraint precision, determinism, scope boundaries). 14 findings total: 1 Major, 13 Minor.

## Findings

### F-1: SKILL.md missing refactor agent invocation

- **File:** `agent-core/skills/orchestrate/SKILL.md`
- **Location:** Absent from execution loop (Sections 3.3-3.5)
- **Axis:** Functional completeness
- **Severity:** Major
- **Design reference:** Architecture > Orchestrate Skill > "What changes from current skill" — "Refactor agent invoked for complexity warnings with deslop directives"
- **Description:** The design lists refactor agent invocation as a change from the current skill, triggered by complexity warnings. The SKILL.md has no reference to the refactor agent anywhere. The corrector agents review quality but don't dispatch to refactor for structural issues. The design's Refactor Agent Updates section adds deslop directives, factorization-before-splitting, and resume pattern to the refactor agent — but the orchestrate skill never invokes it.
- **Impact:** When a corrector or verification step identifies complexity warnings (file too large, duplication), there's no mechanism to dispatch the refactor agent. The orchestrator would need to handle this ad-hoc or ignore it.
- **Recommendation:** Add a refactor dispatch path, likely after corrector review when complexity warnings are surfaced. The corrector report could include a "REFACTOR: [files]" signal that triggers refactor agent dispatch before proceeding.

### F-2: SKILL.md inline phase content source assumes phase-directory layout

- **File:** `agent-core/skills/orchestrate/SKILL.md`
- **Location:** Line 58
- **Axis:** Functional correctness
- **Severity:** Minor
- **Description:** Section 3.0 says "Read the phase content from `plans/<name>/runbook-phase-P.md`." This path only exists when the runbook uses a phase-directory layout (multiple `runbook-phase-*.md` files). For single-file runbooks with inline phases, the content is embedded in the runbook itself, not in a separate `runbook-phase-P.md` file. The orchestrator plan's `## Phase Files` section (generated by `prepare-runbook.py`) only exists for phase-directory runbooks.
- **Impact:** An orchestrator executing a single-file runbook with inline phases would fail to find `runbook-phase-P.md`. However, the orchestrator plan has a `## Phase Files` section that lists the correct paths when phase files exist, and a sonnet orchestrator could likely recover by reading the runbook.
- **Recommendation:** Change to: "Read the phase content from the orchestrator plan's `## Phase Files` section (if present) or from the runbook itself."

### F-3: SKILL.md Section 6 completion diff uses first-ever commit as baseline

- **File:** `agent-core/skills/orchestrate/SKILL.md`
- **Location:** Line 225
- **Axis:** Functional correctness
- **Severity:** Minor
- **Description:** The command `git log --all --oneline | tail -1 | cut -d' ' -f1` gets the repository's first commit, not the runbook execution start point. The resulting diff shows all changes since the beginning of the repository, not changes made during runbook execution.
- **Impact:** The changed files list will be overly broad, potentially including thousands of files unrelated to the runbook. A sonnet orchestrator would likely still function (passing the list to a corrector), but the corrector's review scope would be inflated.
- **Recommendation:** Use a more targeted approach such as tagging the start commit before execution begins, or using `git diff --name-only` against a baseline captured at step 1.

### F-4: SKILL.md Section 6 single-phase completion omits outline reference

- **File:** `agent-core/skills/orchestrate/SKILL.md`
- **Location:** Line 228

- **Severity:** Minor

### F-7: SKILL.md Section 3.5 phase boundary detection instruction imprecise

- **File:** `agent-core/skills/orchestrate/SKILL.md`
- **Location:** Lines 161-162
- **Axis:** Determinism
- **Severity:** Minor
- **Description:** "Read the phase field from the next step entry in the orchestrator plan (pipe-delimited format). If phase changes (or final step):" — The orchestrator plan has explicit `PHASE_BOUNDARY` markers. The instruction should reference the marker, not require the orchestrator to compare phase fields between consecutive entries (which is a derived check when the data is already explicit).
- **Impact:** A sonnet orchestrator can handle both approaches, but the `PHASE_BOUNDARY` marker is the designed mechanism (design line 268: "PHASE_BOUNDARY marker on last step of each phase"). Using the marker is more deterministic.
- **Recommendation:** Change to: "If the current step entry has `PHASE_BOUNDARY` marker (or is the final step):"

### F-8: refactor.md resume pattern diverges from design trigger condition

- **File:** `agent-core/agents/refactor.md`
- **Location:** Line 198
- **Axis:** Conformance
- **Severity:** Minor
- **Design reference:** Architecture > Refactor Agent Updates > Resume pattern
- **Description:** Design specifies the resume trigger as "precommit still has warnings after changes." The refactor.md says "interrupted mid-refactoring." These are different conditions — one is about incomplete quality resolution, the other is about agent interruption. The design also specifies "once" (single resume attempt) and "If resumed and still cannot fix: return error, orchestrator delegates recovery" — both omitted from the refactor.md.
- **Impact:** Without the "once" constraint and the error-return behavior, the orchestrator might resume indefinitely or the agent might not know to report failure after resumed attempt.
- **Recommendation:** Align the trigger condition with the design. Add the "once" constraint and the error-return instruction.

### F-9: refactor.md deslop placement diverges from design

- **File:** `agent-core/agents/refactor.md`
- **Location:** Lines 101-109
- **Axis:** Conformance
- **Severity:** Minor
- **Design reference:** Architecture > Refactor Agent Updates > Deslop directives — "add to Refactoring Protocol, before Step 3"
- **Description:** Design says to add deslop directives "before Step 3" in the Refactoring Protocol. The refactor.md places them as "Step 2b" — a new step between Step 2 (Design Refactoring) and Step 3 (Execute Refactoring). This is functionally correct placement (before execution), but the naming ("Step 2b") suggests it's a sub-step of Step 2 rather than a separate step. The design's intent was that deslop happens before structural execution.
- **Impact:** Minimal — the sequencing is correct. "Step 2b" naming is slightly awkward but unambiguous.
- **Recommendation:** No change needed — placement is functionally correct.
